| `/money/review` | Review queue (§5.4) | Pending bank/receipt transactions w/ AI category suggestion chip (never auto-applied), confirm/override; **suggestions learn** (2026-07-12): a category correction (at confirm or later via PATCH) upserts a per-account `VendorCategoryMemory` keyed on a bank-descriptor-noise-stripped vendor key, future transactions from that vendor get the learned pick at high confidence, and the chip reads "from your past choice"; rent-match chip for income deposits that look like an open expected rent (accepting links the deposit and marks the RentPayment paid); property/unit attribution selects on the confirm card. Bulk "Confirm all suggested" now applies a 0.7 minimum-confidence gate (the income fallback sits at 0.5 by design, so "every income is Rent" can't be mass-confirmed) and also skips items flagged as **possible duplicates** — a confirmed transaction with the same type, exact amount, within ±3 days, vendors agreeing or absent on one side (hand-logged originals, Stripe FC's vendor-less rows); the flag names the match, uses rent-aware copy when it backs a recorded rent payment, and resolves by Dismiss — never auto-merged. A saved income entry matching an open expected rent (same heuristic as the review queue) triggers an explicit "mark rent paid?" prompt — accepting links the transaction to the RentPayment (method `manual`) so the same rent can't be recorded twice. |
| `/rent` | Rent tracker (§5.5) | Period selector, collected/outstanding/progress tiles (partial count shown), per-tenant status with exact days late, individual + bulk reminders (confirm modal, result toasts). **Partial & aggregated payments** (2026-07-12): many deposits (`RentPaymentDeposit`) sum toward one charge — the record-payment modal takes any amount up to the remaining balance (overshoot rejected; below-remaining records a partial), a shortfall shows "Partial — $X of $Y" (still late-flagged past grace; reminders/late-rent insight dun for the *remaining* amount), and the charge reads paid only once deposits cover it. A per-row Deposits modal lists each deposit with an audited Unlink that recomputes the running total (the ledger row survives as an ordinary transaction). Rent-match suggestions target the charge's *remaining* balance, so a second roommate check still matches. Ledger rows backing a deposit can't have amount/date/type edited or be deleted (unlink first) and the Money ledger marks them "applied to rent". **Per-tenant shares + linkage nudge** (2026-07-12): co-tenants carry a `shareCents` (editable in the co-tenants modal; even-split fallback when unset; soft warning on mismatched sums), tracker rows show each co-tenant's share and paid/due derived from deposit attribution (the record modal's "Paid by" select), and unlinked Rent-categorized income that fits exactly one open charge surfaces as an AiSurface "Link to rent?" nudge on the Rent page (`GET /rent/unlinked-deposits`) — never auto-applied, ambiguous matches suppressed. Charges are one-per-unit-per-month: partial-coverage months prorate by days, due dates never predate the lease, and a renewal switchover or termination reconciles the open charges it shortens (blended old+new proration for the switchover month; future months voided; paid/processing rows untouched — all audited). Timing semantics, by design: tracker "collected" is attributed to the rent *period*, while ledger income (dashboard/P&L) is attributed to the *paid date* — July rent paid Aug 3 is July-collected but August income. **Late fees v1** (*added 2026-07-16*): a flat per-lease policy (`Lease.lateFeeCents` override, null = the account default `Account.defaultLateFeeCents`, 0 = none — carried forward on renewal) with totalDue (amount + fee) driving the tile/reminder/matching math everywhere; a late row gets a one-click **Apply fee** (`POST /rent/payments/:id/late-fee`, explicit user action, never automatic) and **Waive** (`DELETE`, blocked once fully collected); a fee-only deposit categorizes as the seeded 'Late Fees' income category on a cash basis (fee income only on collection). **Grace period** (*2026-07-16*): `Account.graceDays` + a calendar-vs-business-days basis (Mon–Fri; holidays out of scope) are configurable in Settings; a charge only reads late once past grace (late-fee eligibility gates on the same check), while "N days late" keeps counting calendar days from the due date. The Paid column shows the newest deposit's date with a "(partial)" qualifier for partially paid charges (`lastDepositAt`; `paidAt` still means fully covered). **Lease-transition correctness** (*2026-07-16*): charge materialization selects leases by date coverage (not just `active` status), so a renewal's not-yet-viewed gap months and unmaterialized switchover months bill the correct blended prorated amount — byte-identical to the switchover reconciliation path — and creating a lease whose dates overlap an existing lease on the unit is rejected (abutting allowed). |

## AI assistant (PRD §9)

- **Global chat drawer** on every screen (docked panel ≥xl, overlay below, full-screen <md); `?chat=open` deep link; context pre-loading from entry points (e.g. Reports). **Clear-conversation** control resets the transcript and starts a fresh session on the next send. When docked (≥xl) the drawer is non-modal — the page beside it stays scrollable and keyboard-reachable; below xl it's a scroll-locking modal with a backdrop.
- **Structured content blocks** rendered in-transcript: `text` (markdown-lite), `chart` (line/bar/donut/sparkline via the app's real chart components, semantic color roles, a11y description + table alternative), `data_table`, `action_card` (buttons executing allowlisted API calls with busy/done states, or in-app navigation), `ask_user_question`.
- **askUserQuestion** (§9.4): model pauses mid-turn with 2–4 options (+ "Other" free text); radiogroup/checkbox UI, keyboard operable; answer resumes the same assistant message; failed answers roll back and allow retry; pause state survives server restart.
- **Agent loop** (backend): Anthropic tool-use loop (max 8 iterations) over the service layer; 19 read + 8 write tools; render tools validated against the shared block schemas; tool activity indicator ("Checking your ledger…"). Write tools are permission-gated per acting member (`deniedWriteTools` maps each to its `MemberPermission` area), so a member can't perform via chat a write their REST guard would block.
- **Deterministic mock mode** (no API key): regex-scripted flows — cash-flow chart, tax pause/resume → Schedule E table + open-report action, late-rent table + send-reminder action, portfolio-summary fallback — executing real tools against the seeded DB. Composer's suggested prompts match these scripts.
- **Auto monthly review** (§5.7/§9.5): daily scheduler generates last month's review (and refreshes rule-based insights) if missing; snapshot archived as a Report.
- **Nightly bank sync + guided post-connect flow** (*added 2026-07-12*): the daily scheduler runs `importFromBank` for every account with a connected bank feed (gated on a connected Integration row so mock-mode accounts never accrete demo rows; cooldown skips are silent, other sync failures are recorded per-account without blocking insights); connecting a bank in Settings or the onboarding wizard chains the first import automatically and the toast deep-links into the review queue; a `transactions_pending_review` insight (rule 6, §4) keeps the dashboard pointing at unreviewed imports until the queue is cleared.
- **Inline insight cards** (§5.1): rule-generated (late rent >5 days, expense spike >125% of trailing avg, renewal window ≤60d, underperforming property <80% of portfolio per-unit avg, contractor cost spike >150% of their prior-job avg), deduped by key so dismissals stick. Surfaced contextually — Dashboard (deck of all active insights under the KPIs in the charts column, highest severity on top, animated cycle controls), Property detail, Rent Collection (late rent), Money (expense spike), Contractors (cost spike) — never on a separate AI page. (The renewal card lives in the Dashboard deck since the standalone tenants list was retired, 2026-07-13.)
- **Executable insight actions** (2026-07-11): each insight carries a structured `action` (chat action-card vocabulary) plus a context-aware deep link (`/rent?period=`, `/money?type&categoryId`, renewal → `/tenants/:id` of the soonest-ending lease's primary tenant). api_call actions (late rent → one-click "Send reminder") execute only if on the web action allowlist — blocked ones render disabled with a visible note; executing marks the insight `actioned` (audited `ai_suggested_user_confirmed`).

## Backend (apps/api)

- **REST API** `/api/v1/*`: properties, units, tenants (full CRUD + soft-archive/restore), leases (create/edit/`GET :id`/terminate/co-tenant add+remove/renewal switchover/renewal draft + mock e-sign), transactions (CRUD, review queue, confirm, receipt scan, bank import), categories, contractors (CRUD incl. website + soft-archive/restore + `GET :id` detail with vendor-match derived job history; list rows carry the same derived usage stats), rent (tracker, record payment, payment link, reminders), reports (library, generate, get, CSV/PDF export, email), insights (list, dismiss, mark-actioned, monthly reviews), dashboard (KPIs, cashflow series, activity, insight), documents (multipart upload w/ pdf/image/docx allowlist + 10 MB cap + per-account rate limit, derived-context list, authed download w/ RFC 5987 filenames, rename/retype, delete), chat (sessions, SSE messages, SSE answer), settings/integrations, team (`GET /team`, invite/revoke, member permission update/remove — all owner-only; `GET /settings/me` for the caller's own role + permissions), onboarding (get derived checklist state; patch start/skip/dismiss), healthz. Every request/response validated by `@hearth/shared` Zod schemas. **Authorization** (multi-user, 2026-07-12): write routes carry `requirePermission(area)`/`requireOwner()` preHandler guards (`lib/authz.ts`) — reads stay open, an owner bypasses, and a demo-mode single operator is treated as owner; members without the grant get `403 forbidden`.
- **Data model**: Account, User (auth identity → account; `role` owner|member + `permissionsJson` = granted `MemberPermission[]`), Invite (pending email invite: account, email, granted permissions, status pending|accepted|revoked — matched to an identity at first login), Property, Unit, Tenant, Lease(+LeaseTenant), Category (IRS Schedule E lines), Transaction (cents, source/status/AI-suggestion fields), RentPayment (period-materialized, due-date days-late), Report (snapshotted dataJson), Insight (dedupeKey), ChatSession/ChatMessage (blocksJson, providerStateJson), Integration, AuditLog, Contractor (directory row; usage stats derived, never stored), OnboardingState (1:1 per account; stores only status + skipped steps — step completion derives from portfolio data), Document (polymorphic single-entity attachment — property/unit/tenant/lease/transaction; bytes behind a StorageAdapter: local filesystem mock locally, Supabase Storage in production when `SUPABASE_SERVICE_ROLE_KEY` is set; a `receipt`-typed transaction document sets `Transaction.receiptUrl`).
- **Derivation rules** (unit-tested against pinned seed constants): rent status/days-late, KPI trends vs same-day-of-month prior window, tax set-aside (current + quarterly target), tenant status precedence, contractor usage stats (vendor-name match, case/whitespace-insensitive), insight rules. All period boundaries (dashboard KPIs/series, monthly review, rent tracker periods, scheduler cadence) respect `Account.timezone` (*added 2026-07-16*, Intl-based helpers in `lib/dates.ts`); fiscal year (`taxYearStartMonth`) remains a no-op, deferred.
- **Reports**: real computed data for Schedule E, P&L, Net Cashflow, Rent Roll, General Ledger, Tenant Ledger; structurally-correct simplified output for the other 7 (maturity-flagged).
- **Audit log** on every money/tenant write with actor attribution: `user`, `ai_suggested_user_confirmed`, `system` (model/MCP/scheduler). Now covers property/unit/tenant/lease create/update/archive/restore/terminate/renew/add_tenant/remove_tenant (REST = `user`).
- **Soft archive**: deleting a property/unit/tenant stamps `archivedAt` (never a hard delete); archived entities are hidden from lists + active-portfolio derivations (occupancy, rent tracker, dashboard money KPIs/activity) but remain resolvable via detail and are restorable. Archiving is blocked (409) while an active lease exists. **Financial/tax reports (Schedule E, P&L, GL) retain an archived property's transactions** for accounting accuracy — only the active-portfolio dashboard drops them.
- **Seeded demo portfolio**: 9 properties / 14 units / $13,695 rent roll; 12 paid + 2 late (Okafor 6d, Park 3d); 6 months of history; KPIs land exactly on the wireframe figures ($8,450 net MTD, 86%, $1,690/$2,700 set-aside). Idempotent reseed refreshes the demo clock.
- **Integration adapters** (interfaces + mock/real swap): Plaid (bank import → review queue) has a real Sandbox implementation (`integrations/real/real-plaid.ts`, cursor-based `/transactions/sync`, AES-256-GCM at-rest token encryption) selected automatically when `PLAID_CLIENT_ID`/`PLAID_SECRET`/`INTEGRATION_ENCRYPTION_KEY` are set, else the mock; Stripe Financial Connections (bank import #2, *added 2026-07-12*) has a real implementation (`integrations/real/real-stripe-fc.ts`, session → Stripe.js modal → per-account `transaction_refresh` cursors, subscribes linked accounts to daily refreshes) selected when `STRIPE_SECRET_KEY`/`STRIPE_PUBLISHABLE_KEY` are set, else the mock — and once any real bank provider is configured, unconfigured providers' mocks are skipped so demo rows can't leak into a real ledger; Stripe (payment links) and Docusign (envelopes) are still mock-only — real implementations are 1:1 swaps behind the same interfaces; outbound email (*added 2026-07-18*) has a real Cloudflare Email Service adapter (`integrations/real/real-cf-email.ts`, REST API) selected when `CLOUDFLARE_EMAIL_API_TOKEN`/`CLOUDFLARE_ACCOUNT_ID`/`EMAIL_FROM` are all set, else the console mock — it carries both the report email-to-accountant path and feedback notifications.
- **Auth**: two modes (deployment plan §4.1). Demo mode (default): single seeded demo account; optional `DEV_BEARER_TOKEN` bearer auth (web client sends it via `VITE_DEV_BEARER_TOKEN`); binds 127.0.0.1 by default. Supabase mode (`SUPABASE_JWT_SECRET` or `SUPABASE_URL` set): every request needs a Supabase Auth JWT; first sight of an identity either **accepts a pending team invite** for its email (joining that account as a `member` with the invited permissions), links to an unclaimed pre-auth account with the same email, or provisions a fresh `Account` + `User` as owner; web login screen appears when `VITE_SUPABASE_URL` + `VITE_SUPABASE_ANON_KEY` are set. The request's resolved role + live permissions gate every write (see Authorization above).
- **Scheduled jobs**: daily monthly-review snapshot + insight refresh run per account (`jobs.service.ts`), via in-process `setInterval` and/or `POST /api/v1/internal/run-daily-jobs` guarded by `CRON_SECRET` (for external cron in deployments that scale to zero).
- **Beta feedback** (*added 2026-07-18*): `POST /feedback` stores a `Feedback` row (category bug/idea/other, message, client-captured `pagePath`, server-captured `userAgent`) and fire-and-forget emails the operator (`FEEDBACK_NOTIFY_EMAIL`; a send failure never fails the write — mirrors `push.service.notifyAccount`). Deliberately ungated (any authenticated member; product feedback isn't a business write — precedent: device registration) and un-audited (not money/tenant-touching); per-account rate limit (`FEEDBACK_RATE_LIMIT_MAX`/min, default 10). No read UI — the operator reads email or the database.
- **Chat hardening** (deployment plan §4.5): per-model-call `aiUsage` structured logs (account/session/message/model/tokens; mock mode reports estimates) and per-account rate limiting on turn-starting chat routes (`CHAT_RATE_LIMIT_MAX`/min, 429 in the ApiError envelope).
- **Production build** (deployment plan §4.4): `npm run build -w apps/api` → esbuild bundle at `dist/server.js`; `apps/api/Dockerfile` (multi-stage, non-root, healthcheck) verified against a postgres:17 container. CI (`.github/workflows/ci.yml`) gates PRs: typecheck → tests → builds → Docker image.
- **Deployed to production** (2026-07-04): https://app.554properties.com — one Cloudflare Worker (`wrangler.jsonc` + `deploy/worker.ts`) serves the web bundle as static assets and routes `/api/*` to the Fastify container; Supabase Postgres (session pooler) + Supabase Auth (JWKS); daily-jobs cron at 09:00 UTC. Merges to `main` auto-deploy via the CI `deploy` job (migrate → wrangler deploy → smoke test).

## MCP server (PRD §10)

- Stdio entrypoint (`npm run mcp -w apps/api`), server name `hearth`, reusing the chat tool registry.

## Cross-cutting

- **Contract package** `@hearth/shared`: all enums, entity/API schemas, chat content-block union, SSE event schemas, cents formatters — single source of truth for both apps.
- **Accessibility (WCAG 2.2 AA target, §7.1)**: skip link, landmarks, `aria-current` nav, focus traps with Esc-returns-focus, radiogroup keyboard navigation, polite live regions (announce on completion, not per token), charts with text alternatives + table views, status never color-only, visible labels + described errors on forms — enforced by axe tests in the web suite.
