## 3. Real integrations (v1 ships mocks behind adapter interfaces)

Interfaces in `apps/api/src/integrations/types.ts`; each swap is one adapter implementation:

- [x] **Plaid** — real Sandbox Link flow + cursor-based `/transactions/sync` into the review queue (`integrations/real/real-plaid.ts`; toggled by `PLAID_CLIENT_ID`/`PLAID_SECRET`/`INTEGRATION_ENCRYPTION_KEY`, account setup §6). Known limitations, deliberately deferred: no webhooks — sync is driven by the manual "Import from bank" button, the post-connect auto-import, and the nightly scheduled sync (*2026-07-12*), so a fresh Sandbox link may still need a follow-up import before Plaid's initial pull finishes; only `added` transactions are processed, `modified`/`removed` are ignored (the `Transaction.externalId` dedup column is in place for a follow-up); Production requires Plaid's app-review process and hasn't been requested.
- [x] **Plaid rent reconciliation**: the Review Queue now computes a rent-match suggestion on load (income bank transaction whose amount exactly equals an open `due`/`processing` `RentPayment` dated within ±14 days of `dueDate`; ambiguous same-rent candidates suppress the suggestion) and renders it as an AiChip ("T. Okafor's Jul rent — never auto-applied"). Accepting it confirms via `POST /transactions/:id/confirm { rentPaymentId }`, which atomically sets the lease's property/unit + Rent category on the transaction and flips the `RentPayment` to `paid` (`method: 'bank'`, `transactionId` linked) with a double-pay guard, audited as `ai_suggested_user_confirmed`. Known accepted misses: early payments for a future period aren't suggested; the chat/MCP `confirm_transaction` tool accepts the same params. The mock Plaid batch includes an income fixture (`plaid_mock_4`, kept in sync with `OKAFOR_RENT_CENTS`) so the flow demos offline.

- [ ] **Transactions gap backlog** (2026-07-12 review of the plan above; real gaps, not yet scheduled — ranked):
  - **Bank-side corrections after confirm are silently dropped**: `applySyncBatch` skips `modified`/`removed` for non-pending rows (deliberate — never rewrite a vouched ledger) but nothing *notifies* either: a pending→posted amount restatement leaves a stale P&L figure, a reversal leaves a phantom row, and a reversed rent-linked deposit is fully stuck (removal skipped, RentPayment stays `paid`, manual edit/delete blocked by the link guards). Needs a "bank changed this after you confirmed it" review surface.
  - **Timezone / fiscal-year settings are stored but never used**: `Account.timezone` and `taxYearStartMonth` are surfaced in Settings but all period math is UTC calendar months (`lib/dates.ts`) — a late-evening transaction for a non-UTC landlord lands in the adjacent month/tax year, and a visible setting that silently does nothing is its own trust break.
  - **Unassigned-transaction inconsistency**: dashboard KPIs include property-less transactions, but NOI-by-property and the monthly review's per-property nets exclude them — surfaces visibly disagree — and nothing nudges assigning a property after confirm (bank imports start unassigned; only Schedule E shows a "Portfolio / unassigned" bucket).
  - **Post-confirm dead ends in the UI**: no transaction delete exists in the web app; the edit modal exposes only property/unit/category, so an amount/date/vendor typo on a confirmed manual entry can't be fixed from the UI (the PATCH API allows it); `dismissed` is terminal (no un-dismiss — a mistaken "Dismiss all" is unrecoverable).
  - **Re-attribution loophole on rent-linked rows**: the update guard blocks amount/date/type but allows category/property/unit — recategorizing a linked deposit away from "Rent" removes it from income reports while the tracker still shows the period paid (the exact Money↔Rent divergence Workstream C targets, via a path it doesn't cover).
  - **Contractor stats ride the vendor string**: `jobsCount`/`avgCostCents`/cost-spike insight derive from case-insensitive vendor matching with no FK — any vendor edit or delete silently rewrites contractor history.
  - Minor: `schema.prisma` `Transaction.status` comment omits `dismissed`; Schedule E export has no receipt/substantiation column (receipt links are polymorphic strings that dangle on hard delete).

## 5. Polish / engineering hygiene

- [ ] **CI pipeline**: run typecheck + both test suites + build + the axe smoke as merge-blocking checks (everything is scripted; just needs a workflow file).
- [ ] **Playwright e2e**: the live end-to-end flow (seed → dashboard → chat pause/resume → action card) was verified manually via curl; codify it as a browser e2e.
- [ ] **Bundle size**: Recharts pushes the main chunk past 500 kB — add `manualChunks` or lazy-load chart components.
- [ ] **Transaction list pagination**: API supports cursor pagination; the UI fetches `limit=100` with no load-more.
- [ ] **"Already reminded" state**: `RentTrackerRow` lacks `remindedAt`, so repeat reminders are only surfaced via the send-result toast; add the field to the shared schema + row.
- [ ] **Session context persistence**: chat session `context` lives inside server-internal `providerStateJson`; give it a real column if session lists ever need to show it.
- [ ] **Clear an optional field on edit**: the property/unit edit modals send `undefined` (omit) for a blanked nickname/notes/market-rent, so the PATCH can't clear a previously-set value — it's silently retained. Needs the `Update*InputSchema` fields to accept `null` and the services to treat `null` as "clear" vs `undefined` as "leave". (QA finding, low; contract change deferred out of the CRUD pass.)
- [ ] **Archived-entity browser**: soft-archive + restore exist end-to-end, but there's no UI to *see* archived properties/tenants/units (restore is only wired via the hook); add an "Archived" filter/section if users need to find and restore them.
